# Zero-Downtime Blue-Green Deployment

## Problem

All services go down together during deploys because `update-server.sh` runs `docker compose down` then `docker compose up -d --build`. Most deploys only change one or two services, but every service takes the hit. There's also no pre-deploy testing beyond grit-provisioner unit tests.

## Solution

Per-service blue-green deployments with a multi-layer testing pipeline. Only changed services restart. New containers are health-checked before receiving traffic. Old containers keep serving until the swap is confirmed.

## Architecture

### Deploy Flow (Per Changed Service)

```
1. Start new container via secondary Compose project (-p blue) with network alias "<service>-blue"
2. Wait for Docker health check to pass (timeout derived from service's health check config)
3. If fails → tear down new container, keep old running, report error
4. If passes → rewrite Caddyfile upstream to "<service>-blue", run `caddy reload`
5. Stop old container via primary Compose project
6. Rewrite Caddyfile upstream back to standard service name
7. Bring service up in primary Compose project (takes over with correct labels)
8. Wait for health check, run `caddy reload` to point to standard name
```

### CI Pipeline (GitHub Actions)

```
Push to main
  → Run unit tests (grit-provisioner server_test.py)
  → Validate compose config (docker compose config)
  → Build Moodle image, push to ghcr.io (public)
  → Deploy to VM (SSH + deploy script)
  → Smoke test public health endpoints
```

Only Moodle has a custom Dockerfile and gets built/pushed to ghcr.io. All other services use upstream images. The generated `docker-compose.yml` on the VM is updated to reference `image: ghcr.io/MakeNashville/moodle:<sha>` instead of `build: ./deploy/moodle`, so both primary and blue Compose projects pull the pre-built image.

### Change Detection

- `update-server.sh` regenerates all config files (Caddyfile, docker-compose.yml, .env) from GCP metadata as it does today
- **After** file regeneration, `deploy.sh` runs `docker compose config` and computes a per-service hash
- Compares against hashes saved from the last successful deploy at `/opt/outline/.deploy-state`
- Only changed services go through blue-green
- Missing state file (first run) triggers a standard `docker compose up -d`

### Service Categories

**Blue-green (zero-downtime):** Outline, Shlink, Shlink-web, OAuth2-proxy, n8n, Moodle, GRIT-provisioner

**Excluded (quick restart or maintenance window):**
- **Caddy** — Binds host ports 80/443, cannot run two instances. Uses `docker compose up -d --no-deps caddy` for sub-second restart.
- **Postgres** — Stateful, rarely changes. Standard restart with brief maintenance window if needed.
- **Redis** — Stateful, rarely changes. Standard restart with brief maintenance window if needed.

If an excluded service (Postgres/Redis) needs updating, it is restarted **before** any blue-green swaps begin.

### Deploy Ordering

1. Database provisioning (create databases if missing) — preserved from current `update-server.sh`
2. Excluded services updated if needed (Postgres → Redis → Caddy)
3. Blue-green services deployed **sequentially**, one at a time, to limit memory pressure on the e2-medium (4 GB RAM)

### Container Management

- New container started via a **secondary Compose project** (`docker compose -p blue -f docker-compose.blue.yml up -d <service>`)
- `docker-compose.blue.yml` is generated by the deploy script from the main `docker-compose.yml`, with:
  - Network aliases added (e.g., `outline-blue`)
  - Attached to the primary project's network
  - `depends_on` stripped (dependencies are already running in the primary project and reachable via shared network)
- This gives proper Compose label management — no `docker run` or manual renaming needed
- After the swap is confirmed, the blue project's container is torn down and the service is brought up under the primary project using `docker compose up -d --no-deps <service>` to avoid triggering Caddy's `depends_on` chain

### Caddy Traffic Swap

1. **Global** find-and-replace of `<service>:<port>` with `<service>-blue:<port>` across the entire Caddyfile (handles services like oauth2-proxy that appear in multiple `reverse_proxy` and `forward_auth` directives)
2. `caddy reload` applies the change gracefully (no dropped connections)
3. Old container stopped, service started under primary project
4. Global find-and-replace back to `<service>:<port>`
5. `caddy reload` again

Two reloads per service, but each is sub-millisecond with no dropped connections.

### Shared Volume Safety

During the brief overlap when both old and new containers are running:
- Caddy still routes all traffic to the **old** container until the reload
- After reload, all traffic goes to the **new** container
- There is no window where both containers serve user requests simultaneously
- The old container is stopped immediately after the Caddy reload confirms
- This means shared volumes (e.g., `moodle_data`, `n8n_data`) are never written to by two active serving instances

### Health Check Timeouts

The deploy script reads each service's health check config from the resolved Compose output and computes the timeout as:

```
timeout = start_period + (retries × interval)
```

For example, Moodle: `start_period(120s) + retries(10) × interval(10s) = 220s`

This prevents the deploy script from killing a healthy but slow-starting container.

### Rollback

- If new container fails health checks, it's killed. Old container was never stopped, so no user impact.
- Failed deploys are reported in CI output and exit with failure code
- No automatic rollback from smoke tests — deploy script already prevented bad containers from receiving traffic

### Smoke Tests

Post-deploy validation from CI:

| Service | Endpoint | Check |
|---|---|---|
| Outline | `https://{DOMAIN}/_health` | HTTP 200 |
| Shlink | `https://go.makenashville.org/rest/health` | HTTP 200 |
| Shlink-web | `https://links.makenashville.org/` | HTTP 200 or 302 |
| OAuth2-proxy | `https://links.makenashville.org/oauth2/ping` | HTTP 200 |
| n8n | `https://automations.makenashville.org/healthz` | HTTP 200 |
| Moodle | `https://learn.makenashville.org/login/index.php` | HTTP 200 |
| GRIT Provisioner | Internal `docker exec` health check | Exit 0 |

- GRIT provisioner has no external Caddy route — validated via `docker exec` on the VM instead of an HTTP request from CI
- Shlink-web is behind OAuth2-proxy — HTTP 200 or 302 (redirect to login) both indicate the service is up; 502 would indicate it's down
- OAuth2-proxy validated via its `/oauth2/ping` endpoint through Caddy
- 3 retries with 10s intervals per endpoint
- CI job fails if any service fails all retries

## Constraints

- Single GCP Compute Engine instance (e2-medium, 4 GB RAM) — sequential blue-green to limit memory pressure during overlap
- Postgres, Redis, and Caddy excluded from blue-green for statefulness/port binding reasons
- ghcr.io used for image registry (public, no auth needed on VM)
- Caddy reload used for traffic swap — graceful, no dropped connections
- Deploy state stored at `/opt/outline/.deploy-state` (consistent with existing working directory)
